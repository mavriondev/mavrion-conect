Você é um engenheiro fullstack sênior. Construa um MVP robusto e extensível de um sistema B2B com SDR + CRM + Matching (oferta x demanda) + Conectores dinâmicos (APIs e Scraping) + Jobs em fila, preparado para hospedar em uma VPS com Docker Compose.

OBJETIVO DO PRODUTO
- Captar leads automaticamente a partir de fontes (APIs e scraping), enriquecer e pontuar (lead scoring), organizar em uma fila SDR, e conduzir no CRM (kanban) até fechamento.
- Ter 2 pipelines separados: (1) Investidores (demanda) (2) Ofertantes/Ativos (oferta).
- Ter módulo de Matching que sugere combinações entre interesses de investidores e ativos cadastrados/enriquecidos.
- Todas as integrações (APIs e scrapers) devem ser configuráveis pelo painel (admin) como “Conectores”, sem precisar alterar código para ligar/desligar ou trocar chaves e parâmetros.
- Scraping deve rodar via workers e fila (Redis + BullMQ), com agendamento recorrente e respeito a rate limit.

STACK E ARQUITETURA (obrigatório)
- Monorepo com:
  1) apps/api (Node.js + TypeScript + NestJS ou Express bem modular)
  2) apps/web (React + TypeScript + Vite + Tailwind + componentes simples)
  3) apps/worker (Node.js + TypeScript para jobs BullMQ; scraping com Playwright quando necessário e Cheerio/axios quando não)
- Banco: PostgreSQL
- ORM: Prisma
- Cache/Fila: Redis + BullMQ
- Autenticação: JWT (access + refresh), com RBAC (admin, manager, sdr)
- Deploy em VPS: Docker Compose para api + web + worker + postgres + redis (+ nginx opcional)
- Documentação da API: OpenAPI/Swagger
- Migrations e seed de dados.

PRINCÍPIOS DO MVP
- Entregar funcionando ponta a ponta: login, painel, conectores, captura/enriquecimento, scoring, fila SDR, CRM kanban, matching.
- Design simples, mas limpo e usável (layout dashboard).
- Sem overengineering: porém com estrutura preparada para escalar.

MODELO MULTI-EMPRESA (mínimo)
- Suportar ao menos 1 organização (tenant) por padrão, mas estrutura no banco deve permitir multi-tenant (org_id em entidades principais).
- Usuários pertencem a uma organização.

MÓDULOS E REQUISITOS FUNCIONAIS

1) CONECTORES (DINÂMICOS) – UI + API
Crie um sistema de conectores configuráveis no painel. Cada conector pode ser:
- Tipo: API ou SCRAPER
- Status: ativo/inativo
- Configuração: JSON (apiKey, baseUrl, headers, rateLimit, parâmetros de busca como CNAE, UF, cidade, keywords, etc)
- Agendamento: “cron” simples ou presets (ex: a cada 6h, diário 02:00)
- Mapeamento de campos: regras de normalização (ex: como mapear resposta para Company/Lead/Asset)
- Logs e histórico de execuções (sucesso/erro, duração, itens processados)

Entidades mínimas:
- Connector (id, org_id, name, type, status, config_json, schedule, last_run_at)
- ConnectorRun (id, connector_id, started_at, finished_at, status, stats_json, error_message)
- RawIngest (id, connector_id, external_id, payload_json, fetched_at, hash_dedupe)

UI do painel de conectores:
- Listar conectores (status, último run, itens)
- Criar/editar conector (form com JSON editor simples)
- Botão “Rodar agora”
- Página de runs/logs do conector

2) INGESTÃO → NORMALIZAÇÃO → DEDUP → ENRIQUECIMENTO
Pipeline padrão:
- Ingestão salva dados brutos (RawIngest)
- Normalização converte para entidades internas (Company, Contact, Lead, Asset, InvestorProfile)
- Deduplicação por CNPJ quando houver; se não, por combinação (nome+cidade+UF) com hash
- Enriquecimento complementa com novas consultas (respeitando cache e rate limit)
- Tudo isso executa via jobs (BullMQ), e não na requisição HTTP do usuário.

3) SDR (CAPTAÇÃO + FILA + LEAD SCORING)
- Motor de busca (jobs agendados) que gera leads conforme filtros configuráveis:
  - região (UF/cidade/raio se houver geo), CNAE, porte, palavras-chave, tipo de ativo
- Lead Scoring configurável por regras (no painel):
  - Ex: +20 se CNAE alvo, +15 se porte médio/grande, +10 se tem imóvel rural, +10 se tem processo minerário, etc.
- Fila SDR:
  - Lista priorizada por score, com filtros e “pegar próximo lead”
  - Estados do lead: new → queued → in_progress → contacted → qualified → disqualified
  - Registro de tentativas de contato e próximos passos

Entidades mínimas:
- Lead (id, org_id, company_id, status, score, score_breakdown_json, source, owner_user_id, last_enriched_at)
- LeadRule (id, org_id, name, enabled, rules_json)  // regras de pontuação
- LeadQueueEvent (id, lead_id, type, payload_json, created_at)

UI SDR:
- Tela “Fila SDR”: cards/tabela com score, empresa, região, CNAE, tags, últimas ações
- Tela “Detalhe do lead”: dados completos, timeline de atividades, botões de ação (criar tarefa, registrar ligação, mover para CRM)

4) CRM (KABAN + EMPRESAS + CONTATOS + NEGÓCIOS + ATIVIDADES)
CRM com boas referências de mercado (HubSpot/Pipedrive/Agendor):
Entidades:
- Company (Account): id, org_id, legal_name, trade_name, cnpj, cnae_principal, cnae_secundarios[], porte, revenue_estimate, website, phones[], emails[], address (logradouro, numero, bairro, cidade, uf, cep), geo (lat,lng), tags[], notes
- Contact: id, org_id, company_id, name, role_title, phone, email, whatsapp, linkedin, tags[], consent_flags_json
- Deal: id, org_id, pipeline_type (INVESTOR | ASSET), stage_id, title, amount_estimate, probability, expected_close_date, owner_user_id, source, description
- PipelineStage: id, org_id, pipeline_type, name, order, color(optional)
- Activity: id, org_id, entity_type (lead/company/contact/deal), entity_id, type (call/email/meeting/whatsapp/note), outcome, description, happened_at, next_step_at, created_by
- Task: id, org_id, title, description, due_at, priority, status, assigned_to, related_entity_type/id

Kanban:
- Pipeline de Investidor: Prospectado → Contato Inicial → Qualificado → Negociação → Proposta → Fechado
- Pipeline de Oferta/Ativo: Prospectado → Contato Inicial → Qualificado → Validação de Documentos → Proposta → Fechado
- Permitir editar estágios (admin)

UI CRM:
- Kanban por pipeline, drag and drop
- Páginas: Empresas, Contatos, Negócios, Tarefas, Atividades
- Timeline no detalhe da empresa/negócio
- Busca global simples

5) MÓDULO DE ATIVOS + INVESTIDORES + MATCHING
Ativos (oferta):
- Asset: id, org_id, type (TERRA | MINA | NEGOCIO), title, description, location (cidade/uf + geo), price_asking, area_ha, docs_status, tags[], attributes_json (ex: cultura, irrigação, licença, teor, etc), linked_company_id (se houver)
Investidor (demanda):
- InvestorProfile: id, org_id, name/company, contact_id opcional, regions_of_interest[], asset_types[], ticket_min/max, preferences_json, tags[]
Matching:
- Rotina que gera sugestões:
  - MatchSuggestion: id, org_id, asset_id, investor_profile_id, score, reasons_json, created_at, status (new/accepted/rejected)
- Score do match baseado em:
  - região, tipo de ativo, faixa de preço, atributos (tags e JSON), status documental

UI Matching:
- Tela com sugestões (ordenadas por score), filtro por tipo e região
- Detalhe mostra “por que deu match” (reasons)

6) SCRAPING (MVP + ARQUITETURA)
- Implementar um “framework” de scraping:
  - Scraper interface padrão: run(connectorConfig) -> array de itens normalizados ou raw payload
  - Suportar Playwright para páginas dinâmicas e Cheerio/axios para estáticas
- No MVP, implemente ao menos:
  - 1 conector API real (ex: BrasilAPI CNPJ ou ReceitaWS) com cache e rate limit
  - 1 scraper de exemplo (site público simples) apenas para demonstrar o pipeline (pode ser um “mock scraper” com HTML local + um modo real opcional)
- Todos scrapers rodam apenas via worker, com jobs e logs.

7) COMPLIANCE BÁSICO (MVP)
- Incluir no sistema:
  - Campos de consentimento e base legal simples no Contact (consent_flags_json)
  - Audit log mínimo (quem alterou e quando) para entidades principais (Company, Contact, Deal)
  - Rate limiting e backoff em conectores
  - Página “Termos/Privacidade” básica no web

REGRAS DE IMPLEMENTAÇÃO
- Não invente integrações que não existam. Quando não der para integrar de verdade no MVP, crie conector “stub” e deixe claro no código como plugar o real.
- Evite hardcode: qualquer fonte deve ser conectada pelo painel.
- Toda captura/enriquecimento/scraping deve ir para fila (BullMQ). A API apenas dispara jobs.
- Garanta que o sistema funciona com dados seed sem depender de fontes externas.
